Migration Guide: Android 2.0.20 → 2.2.0

Breaking Changes​

1. SdkConfig — fields removed​

The following fields have been removed from SdkConfig. The SDK now resolves these values internally.

Before (2.0.20):

val sdkConfig = SdkConfig(
    hostAppApplicationId = packageName,
    hostAppApplicationName = getString(R.string.app_name),
    isLoadFromAsset = false,
    appVersion = getAppVersion(applicationContext),
    osVersion = getOSVersion(),
    deviceId = Toolbox.getUUID(applicationContext),
    imei = Toolbox.getUUID(applicationContext),
    apiKey = "<your-api-key>",
    variant = BuildVariant.Uat,
    partnerKey = "<your-partner-key>",
    secretId = "<your-secret-id>",
)

After (2.2.0):

val sdkConfig = SdkConfig(
    hostAppApplicationName = getString(R.string.app_name),
    isLoadFromAsset = false,
    apiKey = "<your-api-key>",
    variant = BuildVariant.Uat,
    partnerKey = "<your-partner-key>",
    secretId = "<your-secret-id>",
)

Removed fields: hostAppApplicationId, deviceId, imei, osVersion, appVersion.

2. HostView — constructor changed​

Before (2.0.20):

HostView(
    activity = this,               // FragmentActivity
    containerId = R.id.container,
    miniAppHeaderConfig = myHeaderConfig  // MiniAppHeaderConfig?
)

After (2.2.0):

HostView(
    context = this,                // Context
    containerId = R.id.container,
    miniAppHeaderConfig = myThemeConfig   // MiniAppThemeConfig?
)
  • Replace activity with context. The activity property still exists as a nullable derived value (context as? FragmentActivity) for backward-compatible internal use.
  • miniAppHeaderConfig now expects MiniAppThemeConfig instead of MiniAppHeaderConfig.

3. MiniAppConfig.headerConfig renamed to themeConfig​

Before (2.0.20):

MiniAppConfig(
    initRequest = initRequest,
    headerConfig = myHeaderConfig   // MiniAppHeaderConfig?
)

After (2.2.0):

MiniAppConfig(
    initRequest = initRequest,
    themeConfig = myThemeConfig     // MiniAppThemeConfig?
)

4. HostAppUser removed​

The HostAppUser data class has been removed. Remove any references to it in your codebase.

5. SdkException constructor signature changed​

If you subclass or directly construct SdkException, the primary constructor now requires code: String? as a third parameter.

Before:

SdkException(message = "error", cause = e)

After:

SdkException(message = "error", cause = e, code = null)
// or use the two-parameter secondary constructor (unchanged):
SdkException(message = "error", cause = e)

The two-parameter (message, cause) and single-parameter (message) secondary constructors are unchanged.

6. compileSdk bumped to 35​

Update your module's build.gradle.kts:

android {
    compileSdk = 35
}
